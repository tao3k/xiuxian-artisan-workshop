# ADR-005: Adversarial Critique as MemRL Reward Signal

## 1. Context and Problem Statement

Our current `omni-memory` implementation follows the **MemRL** paradigm, where agent performance is improved by reranking episodic memories based on their learned utility (Q-values).

However, the "Reward" ($r$) signal used to update these Q-values is currently disconnected from the "Reasoning" process:

- Reward is usually computed lazily during session consolidation (when turns are drained from the window).
- The reward logic is often generic (e.g., "did the tool return success?").
- In adversarial scenarios like "Agenda Validation", the system already possesses a highly sophisticated critic (the "Strict Teacher"), but its nuanced feedback (the `<score>` tag) is discarded after the flow control decision is made.

We need a way to turn **LLM-driven critique** into **mathematical reinforcement** for the agent's memory.

## 2. Decision

We will integrate the "Strict Teacher's" audit score directly into the MemRL feedback loop as a primary reward signal.

1.  **Immediate Feedback**: When the `LlmAugmentedAuditMechanism` (established in ADR-003) extracts a `<score>` from the critic's response, it will immediately trigger a Q-value update for the preceding "Steward" proposal.
2.  **Reward Mapping**: The XML `<score>` (range 0.0 to 1.0) will be mapped 1:1 to the Reinforcement Learning reward $r$.
3.  **Synaptic Context Extension**: The `ZhenfaContext` will be upgraded to support **Signal Emission**. Tools/Mechanisms can emit "Utility Signals" back to the Orchestrator.
4.  **Episode Tagging**: Every proposal generated by the "Agenda Steward" will be temporarily cached as a "Pending Episode." When the "Strict Teacher" provides a score, the Pending Episode is finalized and committed to `omni-memory` with the provided reward.

## 3. Technical Design & Algorithm

### 3.1 The Q-Learning Update Rule

Upon receiving a score $S$ for an episode $E$:
$$Q(E) \leftarrow Q(E) + \alpha 	imes (S - Q(E))$$
_Where $\alpha$ is the learning rate (e.g., 0.2)._

### 3.2 Signal Emission via `ZhenfaContext` (Pseudocode)

```rust
// In the Strict Teacher Audit Node
async fn execute(...) {
    let critique = call_llm().await;
    let score = parse_xml_score(critique);

    // Emit the reward signal back to the kernel
    ctx.emit_signal(ZhenfaSignal::Reward {
        target_episode_id: ctx.last_turn_id(),
        value: score,
        source: "strict_teacher_critique"
    });
}
```

### 3.3 Two-Phase Retrieval Impact

When the user next asks to "Schedule tasks", the **Phase 2 (Reranking)** of memory recall will:

- **Boost** previous plans that received high scores (e.g., 0.9) from the Teacher.
- **Suppress** previous plans that were "beaten down" (e.g., 0.2) by the Teacher.
- Result: The "Agenda Steward" effectively **learns to avoid procrastination** by recalling the "pain" of past critiques.

## 4. Consequences

### Positive

- **Self-Correcting Memory**: The agent improves its planning quality without any fine-tuning or weight changes.
- **Nuanced Reinforcement**: Moves beyond binary success/failure to a spectrum of "quality" defined by another LLM.
- **Closed-Loop Evolution**: Completes the "Unity of Knowledge and Action" by allowing action-outcomes to reshape knowledge-recall.

### Negative

- **Critic Hallucination Risk**: If the "Strict Teacher" provides an unfair or hallucinated score, it will pollute the Q-table. (Mitigation: Set a low learning rate $\alpha$).
- **State Complexity**: Requires tracking "Pending Episodes" across node executions in the `Qianji` engine.

## 5. Implementation Plan

1.  **Update `omni-memory`**: Ensure the Rust `QTable` can be updated via the `ZhenfaOrchestrator` hooks.
2.  **Extend `ZhenfaContext`**: Implement the signal emission bus.
3.  **Refactor `LlmAugmentedAuditMechanism`**: Add the reward emission logic.
4.  **Verify Evolution**: Run a "Bootcamp" test where an agent is criticized multiple times and observe if its retrieval bias shifts towards higher-quality plans.
