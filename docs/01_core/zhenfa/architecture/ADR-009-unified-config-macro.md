# ADR-009: Unified Config Macro and Conflict Enforcement

## 1. Context and Problem Statement

As the system scales, multiple crates (e.g., `xiuxian-skills`, `omni-agent`, `xiuxian-wendao`) require their own configuration schemas.

We need a model that balances:

- **Developer Sovereignty**: Crate authors should manage their own defaults without cluttering a global file.
- **User Simplicity**: End-users should only deal with a single, consolidated configuration file.

## 2. Decision: Distributed Defaults, Unified Overrides

We will implement a hybrid configuration model driven by the `#[xiuxian_config]` macro.

### 2.1 The Configuration Hierarchy

1.  **Internal Base (Crate-Local)**: Each crate stores its default TOML in `resources/config/` (e.g., `wendao.toml`, `skills.toml`). This is embedded at compile-time.
2.  **User Layer (Consolidated)**: All user-level overrides are merged into a single **`xiuxian.toml`** file at `$PRJ_CONFIG_HOME/xiuxian-artisan-workshop/`.
3.  **Namespace Projection**: The `#[xiuxian_config](namespace = "xxx")` macro maps the `[xxx]` table in the global `xiuxian.toml` to the corresponding crate's configuration struct.

### 2.2 Conflict Enforcement

If a global `xiuxian.toml` exists, the system will actively reject any standalone configuration files (e.g., a separate `skills.toml` in the user's config directory). This ensures a **Single Source of Truth** and prevents configuration drift.

## 3. Technical Design

### 3.1 Macro Attributes

- `namespace`: The table key within `xiuxian.toml`.
- `internal_path`: The path within the crate's resources (e.g., `resources/config/wendao.toml`).
- `orphan_file`: Optional legacy fallback filename. Set `orphan_file = ""` to fully disable orphan scanning.

### 3.2 Loading Logic

The `load()` method generated by the macro will:

1.  Read the embedded default TOML.
2.  Locate `xiuxian.toml` in the workshop configuration home.
3.  Perform a **Deep Merge** of the relevant namespace from the global file into the default values.
4.  Throw a fatal error if an "orphan" config file (matching the namespace) is detected alongside the global file.

### 3.3 Canonical Declaration Pattern

```rust
#[xiuxian_macros::xiuxian_config(
    namespace = "skills",
    internal_path = "resources/config/skills.toml",
    orphan_file = "",
    array_merge = "append"
)]
```

Project policy requires explicit `internal_path` for every crate using `#[xiuxian_config]`.

## 4. Consequences

### Positive

- **Modular Development**: Crate-specific logic remains isolated.
- **Clean UX**: Users manage exactly one file for all settings.
- **Fail-Fast Safety**: Detects and rejects redundant configuration files.

## 5. Implementation Plan

1.  Refactor `xiuxian-config-core` to support namespace-based extraction from a single file.
2.  Update all crates to use `#[xiuxian_config]` with explicit internal TOML paths.
3.  Update documentation to guide users on consolidating their settings.

## 6. User Migration Guide (Consolidation)

1. Keep only `$PRJ_CONFIG_HOME/xiuxian-artisan-workshop/xiuxian.toml`.
2. Move legacy orphan file tables into matching namespaces under `xiuxian.toml`.
3. Remove orphan files such as `skills.toml`, `wendao.toml`, or other standalone per-domain files.
4. Restart the runtime and verify no `RedundantOrphan` error is emitted.
