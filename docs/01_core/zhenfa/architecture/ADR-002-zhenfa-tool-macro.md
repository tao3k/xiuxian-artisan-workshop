# ADR-002: Zhenfa Procedural Macro for Native Tools

## 1. Context and Problem Statement

Following the adoption of **ADR-001 (Native In-Process Tool Registry)**, developers are required to implement the `ZhenfaTool` trait for every domain capability they want to expose to the LLM.

Our code audit of the early `WendaoSearchTool` implementation revealed significant boilerplate:

1. **Manual Struct Definition**: Developers must create an empty struct to hold state.
2. **Manual Trait Implementation**: Writing `impl ZhenfaTool for X`.
3. **Manual Schema Generation**: Hand-writing the `serde_json::json!` block for the `definition()` method. This is highly error-prone; if the Rust struct and the JSON Schema drift, the LLM will hallucinate arguments.
4. **Manual Deserialization**: Writing boilerplate `args.get("...")` or `serde_json::from_value` logic inside `call_native`.

This boilerplate violates the **DRY (Don't Repeat Yourself)** principle and degrades the Developer Experience (DX).

## 2. Decision

We will introduce a Procedural Attribute Macro `#[zhenfa_tool]` to automate the generation of `ZhenfaTool` trait implementations.

We will adopt a **"Doc-as-Code & Type-as-Schema"** philosophy: the Rust function signature, its typed arguments, and its doc-comments will serve as the single source of truth for generating both the runtime execution logic and the LLM JSON Schema.

## 3. Technical Design

We will either extend the existing `xiuxian-macros` crate or create a dedicated `xiuxian-zhenfa-macros` crate.

### 3.1 Required Dependencies

To achieve automatic schema generation, we will introduce `schemars` to the workspace.

- `syn`, `quote`, `proc-macro2`: For AST parsing and code generation.
- `schemars`: For deriving JSON schemas directly from Rust structs.
- `darling` (Optional but recommended): For parsing macro attributes easily.

### 3.2 The Target Developer Experience (DX)

A developer should only need to write the following:

```rust
use xiuxian_zhenfa::zhenfa_tool;
use schemars::JsonSchema;
use serde::Deserialize;

/// Arguments for Wendao Search.
/// The doc-comments here will automatically be included in the LLM schema.
#[derive(Deserialize, JsonSchema)]
pub struct WendaoSearchArgs {
    /// The search query with optional directives
    pub query: String,
    /// Maximum number of hits to return
    pub limit: Option<usize>,
}

#[zhenfa_tool(
    name = "wendao.search",
    description = "Search the Wendao graph index and return stripped XML-Lite records."
)]
pub async fn wendao_search(
    ctx: &ZhenfaContext,
    args: WendaoSearchArgs
) -> Result<String, ZhenfaError> {
    // 1. args are strictly typed! No manual JSON parsing needed.
    // 2. We can directly execute native logic.
    let index = ctx.get_extension::<Arc<LinkGraphIndex>>().unwrap();
    let payload = index.search_planned_payload(&args.query, args.limit.unwrap_or(20), ...);

    Ok(render_xml_lite(&payload))
}
```

### 3.3 How the Macro Works (Under the Hood)

When the compiler encounters `#[zhenfa_tool]`, the procedural macro will parse the `wendao_search` function AST and generate a hidden struct implementation:

```rust
// === CODE GENERATED BY MACRO ===

pub struct WendaoSearchTool;

#[async_trait::async_trait]
impl xiuxian_zhenfa::ZhenfaTool for WendaoSearchTool {
    fn id(&self) -> &str {
        "wendao.search"
    }

    fn definition(&self) -> serde_json::Value {
        // Automatically extracts the schema using `schemars`!
        let mut schema = schemars::schema_for!(WendaoSearchArgs);
        serde_json::json!({
            "name": self.id(),
            "description": "Search the Wendao graph index and return stripped XML-Lite records.",
            "parameters": schema
        })
    }

    async fn call_native(
        &self,
        ctx: &xiuxian_zhenfa::ZhenfaContext,
        args: serde_json::Value
    ) -> Result<String, xiuxian_zhenfa::ZhenfaError> {
        // Automatic parameter validation and deserialization
        let parsed_args: WendaoSearchArgs = serde_json::from_value(args)
            .map_err(|e| xiuxian_zhenfa::ZhenfaError::invalid_arguments(e.to_string()))?;

        // Call the original developer function
        wendao_search(ctx, parsed_args).await
    }
}
```

## 4. State Management Refinement

In `ADR-001`, tools held `Arc<LinkGraphIndex>` inside themselves. With functional macros, the tool becomes stateless (a pure function).

**Solution**: The `ZhenfaContext` must be upgraded to act as a **Type-Map (Dependency Injection Container)**, similar to `axum::Extension`.

- During startup: `orchestrator.insert_extension(Arc::clone(&global_link_graph_index))`.
- Inside the macro function: `ctx.get_extension::<Arc<LinkGraphIndex>>().unwrap()`.

This creates a perfectly clean, stateless functional paradigm for tool authors.

## 5. Implementation Plan

1. Add `schemars` to the workspace dependencies.
2. Create `xiuxian-zhenfa-macros` crate with `proc-macro = true`.
3. Implement the `#[zhenfa_tool]` attribute macro using `syn` to parse the function signature and extract the arguments type.
4. Upgrade `ZhenfaContext` in `xiuxian-zhenfa` to support `http::Extensions` or a similar `TypeMap` for dependency injection.
5. Refactor the existing `WendaoSearchTool` and `QianhuanRenderTool` to use the new macro.
